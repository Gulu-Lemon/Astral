# 关于魔女审判

严格来讲，搜查阶段是正文与“魔女审判”的中间阶段，或者说是“魔女审判”的前置阶段。触发尸体发现通告、进入搜查阶段时，显示时间变为“hh:mm”与当前阶段“剩余时间：xx分钟”两栏，原正文循环不再生效，用以下流程代替。虽然不使用正文循环，但和同样也在历史文本中。



## 搜查阶段：

限时1小时。尸体发现通告/广播/触发搜查环节的时候根据当前状况生成一个开始时间（毕竟正好在整点的概率很小嘛）。

添加功能：证物：在证物界面可以查看已保存证物并添加新证物。已保存证物可以查看详细信息（保存时由模型从玩家视角总结相关信息作为证物的说明）、保存时间等，可以输入物品以添加新证物，但是需要让模型检验：物品在当前场景是否存在？是哪个物品？操作是否可行？添加完成后在叙述文本中生成一句话表示玩家把这样物品带在身上了。玩家取走的物品在环境中消失。证物和物品的区别：证物在审判结束后统一复原/处理，不再归为己有，但物品不会。搜查阶段不能添加物品，只能添加证物，因为“现在更应该关注案件调查”。

不用所有人发现，让一两个NPC进行开场白就行。或者玩家想进行陈词也行。陈词完直接进辩论阶段。

# 审判重构设计方案（2026-06-02 定稿）

> 本方案基于"行动计划(ActionPlan)"系统，NPC 和玩家统一使用分钟级时间精度。

---

## 一、核心新概念：ActionPlan

### 1.1 数据结构

```python
@dataclass
class ActionStep:
    action_type: str              # MOVE / EAT / SEARCH / SOCIALIZE / CONFRONT / REST / GUARD / WATCH / ...
    target_location: str | None   # 目标房间（MOVE、SEARCH 用）
    target_id: str | None          # 目标 NPC（SOCIALIZE、CONFRONT 用）
    duration: int                  # 本步骤耗时（分钟）
    description: str               # 行动描述（LLM生成），如"前往宴会厅吃早饭"
    interruptible: bool = True     # 是否可被打断

@dataclass
class ActionPlan:
    agent_id: str
    steps: list[ActionStep]        # 一次性生成多步（方向 A）
    current_step_idx: int = 0
    step_start_time: int = 0       # 当前步骤开始的 time_minutes
    created_at: int = 0            # 计划生成的 time_minutes
    is_completed: bool = False     # 所有步骤执行完毕
```

### 1.2 ActionPlan 生命周期

```
Agent.plan() 调用 LLM → 返回 ActionStep 列表（3-5 步，覆盖 1-2 小时）
  ↓
GameSession._tick():
  while True:
    for 每个 Agent:
      step = agent.plan.steps[current_step_idx]
      if time_minutes - step_start_time >= step.duration:
        # 当前步骤完成
        执行步骤副作用（位置更新、好感度变更等）
        current_step_idx += 1
        step_start_time = time_minutes
        if current_step_idx >= len(steps):
          # 计划执行完毕 → 调用 plan() 生成下一步计划
          agent.plan()
    
    # 每 tick 检查冲突：
    Arbiter._detect_plan_conflicts(all_plans, world)
      → 检测到冲突时触发打断/协商逻辑
    
    time_minutes += 1  # 每 tick 推进 1 分钟
```

### 1.3 时间估算

每次调用 LLM 生成 ActionPlan 时，prompt 要求 LLM 为每个 ActionStep 估算 `duration`（分钟）：

```
Agent {name} 当前状况：
- 位置：{location}
- 时间：第{day}天 {time}（{time_minutes}分钟）
- 情绪：{emotional_state}
- 饥饿程度：{hunger_level}
- 当前计划：{current_plan}

请为 {name} 制定接下来 1-2 小时的行动计划。
每步必须包含：行动类型、目标地点/目标人物、预计耗时（分钟）、简短描述。

你的计划必须考虑：
1. 现在是第几天的什么时间——该吃早饭/午饭/晚饭/休息了
2. 当前情绪状态——焦虑的人可能坐不住
3. 与其他 NPC 的关系——想见谁、想躲谁
4. 已知的事件——广播通知、尸体发现等

输出 JSON:
{
  "steps": [
    {"action_type":"MOVE","target_location":"宴会厅","target_id":null,"duration":10,"description":"前往宴会厅"},
    {"action_type":"EAT","target_location":"宴会厅","target_id":null,"duration":45,"description":"在宴会厅吃早饭"},
    {"action_type":"SOCIALIZE","target_location":"宴会厅","target_id":"No.03","duration":20,"description":"和 No.03 聊天"}
  ]
}
```

## 一点五、NPC 感知平等原则：抹除"玩家"身份标识

### 核心原则

所有 LLM 提示词中，玩家作为 **普通角色 No.13** 出现——NPC 引擎不知道谁是玩家、谁是 NPC。任何人都是"住在同一场景的另一名角色"。

### 双层架构

```
引擎层（内部）                     LLM 提示词层（外表）
─────────────────                   ─────────────────
agent_id = "player"       →         "No.13"
npc_locations["player"]   →         查询时转换为 No.13
affection_map["player"]  →         affection_map["No.13"]
player_met_npcs.add(aid) →         被看见的记录用 No.13
```

引擎层保持 `"player"` 不动（`agent_states["player"]`、`affection_map["player"]`、存档等几十处引用，改发动全身）。只在提示词构建时做一层翻译。

### 改动位置

| 文件 | 函数 | 场景 |
|------|------|------|
| `agent_engine.py:125-203` | `_build_decision_prompt()` | nearby_names 构建 → `"player"` → `"No.13"` |
| `agent_engine.py:132-143` | `_build_decision_prompt()` | aff_str 构建 → `"player"` → `"No.13"` |
| `agent_engine.py:213-251` | `dialogue()` | speaker_id → 调用方传入时已转换 |
| `agent_engine.py:29-63` | `perceive()` | nearby_npcs 构建，`"player"` → `"No.13"` |
| `gm.py:57-63` | `_build_context()` | nearby_names 用 `_npc_label()` 查字符字典，玩家名需同样纳入 |
| `gm.py:80-` | `_generate_narrative_and_options()` | rulings_desc 中 actor 标签需翻译 |

### 效果对比

**改前**（Agent 看到的 prompt）：
```
附近角色：七海澪(No.01), 赤城莉莎(No.08), 玩家(player)
好感：小坂爱莲(player):40(相识，友善)
```

**改后**：
```
附近角色：七海澪(No.01), 赤城莉莎(No.08), 小坂爱莲(No.13)
好感：小坂爱莲(No.13):40(相识，友善)
```

LLM 从编号、名称、格式均无法推断谁是玩家。No.13 就是 No.13。

### 注意事项

- `Intent.target_id` 返回时如果指向 `"No.13"`，需在引擎层反向翻译回 `"player"` 以便 `_apply_rulings()` 正确处理
- GM 叙事中 `_npc_label()` 当前对玩家返回空——需加玩家名回退
- 前端 NPC 面板不受影响（前端用 `S.npcData`，独立渲染逻辑）

## 二、整体数据流

### 2.1 正常模式（玩家行动触发推进）

核心原则：**详略得当、张弛有度**——玩家活跃时详细叙事，无事发生时快进略写。

```
玩家点击某一行动（探索/调查/推进/自由输入）：
  ↓
1. 计时（_tick 循环）：
   while time_minutes < player_action_tick:
     for each Agent:
       执行当前 ActionStep（按 duration 推进分钟）
       步骤完成 → 位置/状态变更 → 下一步
       计划耗尽 → Agent.plan() 生成新计划
       冲突检测（NPC 同室对峙/社交）→ 缓存冲突事件
     time_minutes += 1

   广播检查：
     if 跨越 7:00 / 12:00 / 18:00 / 22:00 → 触发广播事件
  ↓
2. 玩家行动执行：
   · 探索房间 / 调查物品 / 与 NPC 对话 / 自由输入
   · 与 NPC 对话不立刻推进时间，等到本次行动结束时统一估算
   · 玩家行动改变环境 → 所受影响的 NPC 标记 plan.interrupted
  ↓
3. 仲裁：
   · 被打断的 Agent + 有缓存冲突的 Agent → Arbiter 裁决
   · 无事发生的 Agent → 继续当前 ActionStep
  ↓
4. 时间估算：
   · LLM 根据玩家行动 + 冲突事件的叙事，估算本轮耗时（分钟）
   · time_minutes += 估算分钟数
  ↓
5. GM 叙事合成：
   · 基于这段时间内的全部事件（Agent 行动 + 仲裁冲突 + 玩家行动）
   · 生成文学叙事 + 4 个选项
  ↓
6. SSE 推送叙事 + 选项 → 前端

LLM 调用时机（tick 循环不逐分钟调用 LLM）：
  - Agent.plan()：计划耗尽或被中断时
  - GM 叙事合成：每轮玩家行动后
  - 冲突仲裁：NPC 之间有互动时
  - 时间估算：随 GM 叙事/行动响应附带
```

### 2.2 审判触发：全局 ActionPlan 中断

```
尸体发现广播 (discovered_by ≥ 2)：
  ↓
1. 所有 Agent.plan.is_interrupted = True
2. 所有 Agent.plan(replan=True, context="尸体被发现，进入搜查阶段")
   → 生成审判专用 ActionPlan（SEARCH / INTERROGATE / GUARD / WATCH）
3. 正常模式暂停，切换审判流程
```

### 2.3 审判模式

```
┌─ 搜查阶段 (限时 60 分钟) ──────────────────┐
│  玩家调查 + 证物系统（JSON API）             │
│  NPC 按审判 ActionPlan 并行调查             │
│  玩家点"结束搜查"或时间耗尽自动推进           │
└──────────────────────────────────────────┘
  ↓
┌─ 陈述阶段 (1-2 轮，JSON API) ─────────────┐
│  NPC 开场白（LLM 生成）                    │
│  玩家可选补充陈词或直接进辩论               │
└──────────────────────────────────────────┘
  ↓
┌─ 辩论阶段 (限时 60 分钟，SSE 流式) ────────┐
│  LLM 叙事流：NPC 七嘴八舌 + 单人论述交替    │
│  共识出现 → 添加「提议投票」选项（可忽略）    │
│  共识可被推翻，辩论可继续                   │
│  时间耗尽 → 强制进入投票                    │
└──────────────────────────────────────────┘
  ↓
┌─ 投票阶段 (JSON API) ─────────────────────┐
│  玩家手动触发投票（或时间耗尽自动触发）       │
│  玩家 + 存活 NPC 投票 → defendant          │
│  场景可定制规则                            │
└──────────────────────────────────────────┘
  ↓
┌─ 处刑阶段 (JSON API) ─────────────────────┐
│  LLM 生成处刑叙事 + 标记死亡                │
│  trial.active = False，场景可定制规则       │
└──────────────────────────────────────────┘
  ↓
恢复正常模式（所有 Agent 重新调度 ActionPlan）
```


## 四、搜查阶段详细设计

### 4.1 触发条件

### 4.2 NPC 行为（基于 ActionPlan）

Agent 进入审判模式，原有的常规 ActionPlan 被覆盖为审判专用计划。

**新增 ActionType**：
```python
SEARCH = "search"            # 搜索房间/寻找证据
INTERROGATE = "interrogate"  # 询问其他 NPC
GUARD = "guard"              # 看守尸体/保护现场
WATCH = "watch"              # 观察其他人
```

**NPC 角色分配**（LLM 根据性格自动选择）：
- 与被害者关系近 → `SEARCH` 优先
- 性格冷静/侦探型 → `SEARCH` / `INTERROGATE`
- 与被害者有矛盾 → `WATCH` 或 `GUARD`
- 怕事/自我中心 → 可能不参与调查，继续日常计划

**NPC 调查结果存储**：
```python
# AgentState 新增
investigation_result: list[str] = field(default_factory=list)
# 格式：["在死者房间发现了沾血的匕首", "目击了 No.03 昨晚从死者房间出来"]
```

### 4.3 证物系统

**新数据结构**：
```python
@dataclass
class EvidenceItem:
    item_id: str = field(default_factory=lambda: uuid.uuid4().hex[:8])
    name: str = ""                     # 证物名
    description: str = ""              # 详细描述（玩家视角）
    found_by: str = ""                 # 发现者（"player" 或 NPC id）
    location: str = ""                 # 发现地点
    found_time_tick: int = 0           # 发现时的 global_tick
    is_key_evidence: bool = False      # 关键证物
```

**证物添加流程**（带 LLM 校验）：
```
玩家输入 → "我要把桌上那把刀作为证物收起来"
  ↓
LLM 模型校验（3 步判断）：
  1. 这个物品在当前场景中是否存在？
  2. 它具体是哪个物品/对象？
  3. 玩家能否取走它？（物理合理吗？）
  ↓
校验通过 → 生成 EvidenceItem → 加入 case_evidence
            → 从 room_item_state 移除
            → 叙事："小坂爱莲将桌上的匕首小心包好。"
校验不通过 → "那把刀嵌在墙里，她拔不出来。"
```

NPC 发现的证物对所有参与者可见，标注发现者。

### 6.2 SSE 事件序列

```
debate_start    → {}                              # 辩论开始
narrative_chunk → { text: "柳和奈站起身..." }       # 逐段推送叙事
debate_done     → {                               # 本轮完成
  text: "完整文本",
  elapsed_minutes: 8,
  timer_remaining: 34,
  consensus_reached: false,
  consensus_target: null,      # 共识指向的 suspect_id（如果有）
  consensus_ratio: 0,          # 共识支持率（0-1）
  options: [
    {"label":"质疑柳和奈","type":"challenge","target":"No.04"},
    {"label":"出示沾血的匕首","type":"present_evidence","target":"evidence_xxx"},
    {"label":"提出推理","type":"reason","target":null},
    {"label":"静观其变","type":"wait","target":null}
    // 如果 consensus_reached == true，额外追加：
    // {"label":"「提议进入投票」大家似乎都指向了No.03...","type":"suggest_vote","target":"No.03"}
  ],
  npc_opinions: {...}
}
```

### 6.3 玩家选择 → 下一轮

```
玩家选选项 → POST /api/trial/debate_option
  { choice: {type, target}, argument: "可选推理文本" }

后端接收：
  · 更新辩论上下文（记录玩家的质疑/出示/推理）
  · 调节 npc_opinions 权重（出示证物 → +0.3 指向相关 NPC）
  · 返回确认 → 前端重新连接 /api/trial/debate_stream（下一轮）
```

### 6.4 共识机制：手动、可选、可逆

共识存在时**不自动推进**投票——仅添加一个选项。

```
无共识时（4 个常规选项）：
  A. 质疑某 NPC  B. 出示某证物  C. 提出推理  D. 静观其变

有共识时（5 个选项，末尾追加）：
  A. 质疑某 NPC  B. 出示某证物  C. 提出推理  
  D. 静观其变  E. 「提议进入投票」
  
玩家选 A/B/C/D → 继续辩论，共识可能被推翻（比例波动）
玩家选 E → 进入投票阶段
```

**共识判定 = LLM 建议 + 后端算法验证**：
```python
if llm_consensus_reached:  # LLM 在 debate_done 中返回
    backend_check = _backend_consensus_check(npc_opinions)  # >= 2/3 指向同一人
    if backend_check:
        options.append({ type: "suggest_vote", target: consensus_target })
    # LLM 一个人自说自话不算数，后端验证才上选项
```

**唯一强制触发投票**：辩论时间耗尽（timer_remaining ≤ 0）。

# 结局系统设计方案（2026-06-02 实施）

## 数据结构

```python
# state.py
EndingBranch   — ending_id / label / category / condition / narrative_hint
EndingConfig   — trigger_type / trigger_value / revelation_hint / branches
WorldState     — ending_triggered / ending_chosen / ending_resolved / player_is_murderer
```

## 触发机制

```
每轮 run_round() 末尾 → _check_ending_trigger()：

1. 玩家死亡 → 自动死亡结局（不显示选项）
2. survivor_count 触发（幸存 ≤ N）→ 根据条件筛选可选分支
3. location_reached 触发（到达某房间）→ 根据条件筛选可选分支
```

## 条件表达式

| 条件 | 含义 |
|------|------|
| `""` (空) | 始终可用 |
| `player_is_murderer` | 玩家是凶手 |
| `player_not_murderer` | 玩家非凶手 |
| `player_dead` | 自动死亡结局（不显示选项） |
| `trial_executed_wrong` | 审判处刑了错误的人 |

## 云端假期结局矩阵

```
幸存 ≤ 5 + 玩家存活：
  玩家非凶手 ──→ A(永远梦境) / B(面对现实)
  玩家是凶手 ──→ C(魔女诞生) / D(只剩一人)

云端假期特殊规则：指认错误 → 凶手外全员抹杀（_trial_execution 已实现）

自动结局：
  玩家死亡 ──→ 死亡画面（不写外界现实）
```

## 天际迷宫结局

```
到达引航台 + 玩家存活 → 回休眠舱 / 留飞船上
玩家死亡 → 死亡结局
凶手存活不影响触发
```
